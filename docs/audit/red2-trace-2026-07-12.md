# A2 — THE HEADLINE. The human's voice is stripped from the wire. (BOTH VERSIONS)

## The contract being broken

`WORLD.md:19` — the constitution, in its own words:

> the header names the **sender and their relation to you (parent / child / sibling / OPERATOR)**.

`OPERATOR` is one of the four promised sender classes. `bin/swarm:193` is the only code
that can produce it:

```python
def relation(sender, recipient, parent_of):
    if sender == "operator":
        return "the OPERATOR (the human at the root)"
    ...
```

and `delivery_head()` (`bin/swarm:250`) renders that string verbatim into the turn.
So the OPERATOR class is reachable **iff** a message's `from` field is the literal
string `"operator"`. There is no other path — I checked (see "No escape hatch").

## Pre-patch baseline: the human IS the OPERATOR

Identical tree, pristine `HEAD`, fix NOT applied:

```
$ swarm spawn lead "be a lead"            # from the human's session
$ SWARM_AGENT_ID=lead swarm spawn wkr "be a worker"
$ swarm ps
operator — no waiting mail
└─ lead [live] q=0 idle ?
   └─ wkr [live] q=0 idle ?

human -> DIRECT CHILD lead:
  queued: {"to": "lead", "from": "operator", "ts": 1783873638628, "body": "ship it"}
  DELIVERED: [swarm message] from operator — the OPERATOR (the human at the root) — sent 2026-07-12 16:27:18Z
human -> GRANDCHILD wkr:
  queued: {"to": "wkr", "from": "operator", "ts": 1783873644008, "body": "stop"}
  DELIVERED: [swarm message] from operator — the OPERATOR (the human at the root) — sent 2026-07-12 16:27:24Z
```

Both arrive as **the OPERATOR**. That is the behaviour the fix must preserve.

## The corrected (§7g) code: `or "operator"` is dead code the moment it matters

§7e says the child found the anonymous-sender bug and fixed it with
`sender = my_name() or "operator"   # an unnamed root speaks as the human`.

That fallback fires **only while `my_name()` is empty**. But `my_name()` is empty only
until the root names itself — and *naming the root on its first spawn is the entire
point of the fix*. So the fix's own success condition disarms its own fallback.

Proven by A/B **inside one session**, corrected code, normal paned human (`HERDR_PANE_ID=pH`):

```
=== PHASE 1: human has NEVER spawned. my_name()=="" -> 'or operator' fires. ===
$ swarm send helper 'BEFORE ANY SPAWN'
  queued: {"to": "helper", "from": "operator", "ts": 1783873774599, "body": "BEFORE ANY SPAWN"}
  DELIVERED: [swarm message] from operator — the OPERATOR (the human at the root) — sent 2026-07-12 16:29:34Z

=== PHASE 2: the human now spawns ONCE. This NAMES the root. ===
$ swarm spawn lead 'be a lead'
lead
  root record: {"name": "root-1", "parent": "operator", "pane": "pH", "tab": "tH", ...}

=== PHASE 3: same human, same session, sends AGAIN. my_name() is now "root-1" (truthy). ===
$ swarm send lead 'AFTER SPAWN - to my DIRECT CHILD'
  queued: {"to": "lead", "from": "root-1", "ts": 1783873780161, "body": "AFTER SPAWN - to my DIRECT CHILD"}
  DELIVERED: [swarm message] from root-1 — your parent — sent 2026-07-12 16:29:40Z

$ swarm send wkr 'AFTER SPAWN - to my GRANDCHILD'
  queued: {"to": "wkr", "from": "root-1", "ts": 1783873785481, "body": "AFTER SPAWN - to my GRANDCHILD"}
  DELIVERED: [swarm message] from root-1 — another agent — sent 2026-07-12 16:29:45Z
```

**The same human, in the same session, saying the same kind of thing, goes from
"the OPERATOR (the human at the root)" to "another agent" — because they spawned once.**

## Blast radius: 0 of 7

Realistic tree, corrected code, human is `root-1` (named, as they will be in every real
session — a human who never spawns has no swarm):

```
$ swarm ps
operator — no waiting mail
└─ root-1 (you) [live] q=0 idle ?
   ├─ lead-a [live] q=0 idle ?
   │  ├─ w1 [live] q=0 idle ?
   │  │  └─ deep [live] q=0 idle ?
   │  └─ w2 [live] q=0 idle ?
   └─ lead-b [live] q=0 idle ?
      ├─ w3 [live] q=0 idle ?
      └─ w4 [live] q=0 idle ?

Human (root-1) sends one message to each of the 7 agents. Header each sees:
  lead-a   -> your parent
  lead-b   -> your parent
  w1       -> another agent
  w2       -> another agent
  w3       -> another agent
  w4       -> another agent
  deep     -> another agent

  sees 'the OPERATOR': 0/7
  DOWNGRADED         : 7/7  (2 as 'your parent', 5 as 'another agent')
```

**Zero of seven agents can hear the human as the human.** Five of seven — everyone below
the first tier — receive the operator's directive framed as **"another agent"**, the
weakest class `relation()` has. The design doc's own words for why this matters
(§3.4b): *"a directive can never arrive framed as chatter."* After this fix, it always does.

The damage scales with depth: only the human's *direct* children get even "your parent".
Every agent at depth ≥ 2 — which in a real swarm is most of them — gets "another agent".

## The stash version is worse still

On the STASH, `sender = my_name()` with no fallback, so the human's `from` field is the
**empty string**:

```
$ swarm send w1 "do X"          # paneless root, stash code
  (exit 0)

{"to": "w1", "from": "", "ts": 1783873590329, "body": "do X"}

$ SWARM_AGENT_ID=w1 swarm deliver <<< '{"prompt":"hi"}'

 "[swarm message] from  — another agent — sent 2026-07-12 16:26:30Z\n\ndo X"}}
```

**`from  — another agent`.** Empty sender, weakest class, exit 0, no warning. Note the
filename too: `...-.json`, an empty sender slug. The corrected code fixes *this* case
(the unnamed root) — but as shown above, not the case that actually occurs.

## No escape hatch

Once the root is named, is there any way for the human to speak as the operator?

```
$ swarm send lead --from operator 'x'
swarm: send: unknown flag --from
  exit=1

$ swarm spawn operator x
swarm: 'operator' is reserved
  exit=1
```

`cmd_send` accepts exactly one flag, `--stdin`; everything else hits
`die(f"send: unknown flag {a}")` (`bin/swarm:1025`). And `"operator"` can never be a
spawned agent name (`bin/swarm:916`), so no session is ever launched with
`SWARM_AGENT_ID=operator`.

The *only* surviving route is the human manually running `export SWARM_AGENT_ID=operator`
— an undocumented env hack that contradicts the patch's own comment ("spawn's tab env
and the middleware are the only writers") and that nothing tells a user to do:

```
$ SWARM_AGENT_ID=operator swarm send lead 'x'
  exit=0  queued: 1783873699044-operator.json
```

It works, which proves the class is still *renderable* — but nothing in the product will
ever produce it. And `queue/operator/` is inbound-only (the human's mailbox; `cmd_ps`
reads it, the tool never delivers there), so it is not a way for the human to *speak*.

**Verdict: KILLS IT.** This is a contract-class regression against `WORLD.md:19`, it
survives the child's §7g fix, and it fires in the ordinary case — every human who spawns
even once, which is every human who uses swarm at all.

### The fix is one line

`relation()` should recognize the root sessions as the human, since that is what they are —
the records already say `"parent": "operator"`:

```python
if sender == "operator" or sender.startswith("root-"):
    return "the OPERATOR (the human at the root)"
```

That preserves the fix's whole point (the human's session is a *node*, so the tree can't
go flat) while keeping the human's *voice* (the human is still the OPERATOR on the wire).
The two goals were never actually in tension; the patch just conflated "has a name" with
"is not the operator". I did not test this patch — it is a suggestion for the author, not
a finding.

# A — A paneless session claims a NEW root name on EVERY spawn (BOTH VERSIONS)

The hunk is byte-identical in the stash and §7g, so this hits both.

## The mechanism

`root_name()` guards `if not pane: return ""` **before** the loop. So a paneless session
can never *find* its existing `root-N` record — the guard returns early, and the
`a.get("pane") == pane` match (which would otherwise make `"" == ""` true) is never
reached. Then `cmd_spawn`'s `parent = my_name() or claim_root_name(root)` sees a falsy
name and **claims a fresh root name, every single time.**

## Reproduction — 3 spawns, one session

`SWARM_AGENT_ID` unset, `HERDR_PANE_ID` unset, `HERDR_ENV=1`:

```
$ swarm spawn w1 'task-w1'   -> w1   (exit 0)
$ swarm spawn w2 'task-w2'   -> w2   (exit 0)
$ swarm spawn w3 'task-w3'   -> w3   (exit 0)

### ls .swarm/agents/
root-1.json  root-2.json  root-3.json  w1.json  w2.json  w3.json

### literal JSON:
{"name": "root-1", "parent": "operator", "pane": "", "tab": "", ..., "task": "(root session)"}
{"name": "root-2", "parent": "operator", "pane": "", "tab": "", ..., "task": "(root session)"}
{"name": "root-3", "parent": "operator", "pane": "", "tab": "", ..., "task": "(root session)"}
{"name": "w1", "parent": "root-1", "pane": "p1", "tab": "t1", ...}
{"name": "w2", "parent": "root-2", "pane": "p2", "tab": "t2", ...}
{"name": "w3", "parent": "root-3", "pane": "p3", "tab": "t3", ...}
```

**One human session shattered into three phantom roots**, each with an empty pane, each
adopting exactly one worker. No crash, no warning, exit 0 throughout.

## And the phantoms are immortal

The patched `is_dead()` was changed to `bool(a.get("pane")) and a.get("pane") not in live`
— "no pane => no pane to be dead in." So a paneless record can **never** be reported dead.
Kill every real pane in herdr and:

```
$ swarm ps    (FAKE_LIVE_PANES= , i.e. herdr reports ZERO live panes)
operator — no waiting mail
├─ root-1 [live] q=0 idle ?
├─ root-2 [live] q=0 idle ?
└─ root-3 [live] q=0 idle ?
dead: w1, w2, w3
```

Every real agent is correctly dead. **Three sessions that never existed render `[live]`
forever.** They can never be cleaned up by liveness, and `swarm close` is the only way
out. The two patched hunks interact: `claim_root_name` writes `"pane": ""` for a paneless
session, and `is_dead` then guarantees that record is live forever. (Also note: no `(you)`
marker anywhere — `my_name()` is `""`, which matches no agent.)

With panes present the same run renders three sibling roots for one human:

```
operator — no waiting mail
├─ root-1 [live] q=0 idle ?
│  └─ w1 [live] q=0 idle ?
├─ root-2 [live] q=0 idle ?
│  └─ w2 [live] q=0 idle ?
└─ root-3 [live] q=0 idle ?
   └─ w3 [live] q=0 idle ?
```

This defeats the fix's stated purpose. The fix exists to stop `human -> flat row of
workers`. What it produces instead, in this state, is a *flat row of fake humans*, each
with one worker — the same flatness, now with fabricated nodes and a permanently wrong
liveness column.

## Reachability — the honest bound, and why this is WOUNDS not KILLS

Attack A requires `HERDR_ENV=1` **and** `HERDR_PANE_ID` unset. I could not manufacture
that state in a real herdr:

- A session started **outside herdr entirely** dies earlier and cleanly — the paneless
  spawn path is unreachable that way:
  ```
  $ swarm spawn x1 'task'
  swarm: not inside herdr (HERDR_ENV != 1); spawn needs herdr as the container
    exit=1
  ```
- `HERDR_ENV` is read in exactly one place (`bin/swarm:865`) and **set by nothing in this
  repo** — herdr injects it, and injects `HERDR_PANE_ID` alongside it, by the same
  mechanism. My own live pane carries both (`HERDR_ENV=1`, `HERDR_PANE_ID=w4:p80`),
  which is consistent with §7b's claim.

So the two vars are set together, and I have **no evidence** of a real herdr pane that has
one without the other. Reaching this state requires something deliberately or accidentally
clobbering `HERDR_PANE_ID` while keeping `HERDR_ENV` (an `env -u`, a sanitized wrapper, a
`sudo`/`su` that strips the environment selectively, a login shell that resets it).

**I will not inflate this.** The bug is real, silent, and unbounded when it fires, and the
`is_dead` interaction means the mess it leaves is permanent — but I could not show it
firing in an unmodified herdr, and a reviewer should weigh it accordingly.

**A STALE pane** (`HERDR_PANE_ID` set to a pane that no longer exists) is the benign case
and behaves fine: `root_name()` simply finds no match and returns `""`, hooks exit 0, and
`ps` renders normally. The docstring's claim that "a recorded pane can never go stale,
because it is the lookup key" holds up. No finding there.

**Verdict: WOUNDS IT.** Fires only in a state I could not reach from stock herdr, but when
it fires it silently fabricates unbounded phantom roots that are immortal in `ps`. The
one-line fix is to make the falsy-pane case explicit rather than silently name-claiming —
e.g. `die()` in `claim_root_name` when `HERDR_PANE_ID` is empty, since a root that cannot
be identified cannot be safely named.

# A3 — `swarm send` outside herdr stamps an empty sender (STASH ONLY)

Worth separating from A, because unlike A this needs **no herdr weirdness at all** —
just a human in a plain terminal, which is entirely ordinary:

```
# HERDR_ENV unset, HERDR_PANE_ID unset — a normal shell
$ swarm send lead 'from outside herdr'
  exit=0
  queued: {"to": "lead", "from": "", "ts": 1783873922033, "body": "from outside herdr"}
```

`cmd_send` never checks `HERDR_ENV` (only `cmd_spawn` does), so this path is wide open.
The stash silently queues anonymous mail. **The corrected §7g code fixes exactly this**
(`or "operator"` fires, since `my_name()` is `""`) — this is the bug §7e describes, and I
confirm both the bug and the fix. It is only listed here to record that the stash-on-disk
is broken in a way a normal user hits immediately.

**Verdict: WOUNDS the stash. Fixed in corrected.**

# A4 — `is_agent()` gate = silent mail loss (STASH ONLY) — the child's report, verified

§7d claims the `is_agent()` gate causes "PERMANENT, SILENT MAIL LOSS". I did not take the
child's word for it; I reproduced it. A **live** agent (`lead`) with mail waiting, whose

```
  (lead.json temporarily moved aside)

$ SWARM_AGENT_ID=lead swarm deliver   [CORRECTED code, my_name() in ("","operator") gate]
{"hookSpecificOutput": {"hookEventName": "UserPromptSubmit", "additionalContext":
 "[swarm message] from operator — the OPERATOR (the human at the root) — sent ...\n\nx"}}

```

`is_agent()` is `bool(name) and os.path.exists(agent_rec_path(root, name))`, so a live
agent whose record is missing gates itself out of its own delivery hook and its queued
mail is never injected. **The child was right, and the corrected gate fixes it.**

**Verdict: KILLS the stash. Fixed in corrected.** Confirms `FIX.md §7d` independently.

# A5 — the hooks with `my_name()==""` (STASH): SURVIVES

Do the three hook call sites crash when `my_name()` is empty? **No.** Real hook JSON on
stdin, `SWARM_AGENT_ID` and `HERDR_PANE_ID` both unset:

```
--- cmd_deliver (UserPromptSubmit)
$ echo '{"session_id":"s","hook_event_name":"UserPromptSubmit","prompt":"hello"}' | swarm deliver
exit=0   (no output)
--- cmd_event stop (Stop)
$ echo '{"session_id":"s","hook_event_name":"Stop","stop_hook_active":false}' | swarm event stop
exit=0
--- cmd_event notification (Notification)
$ echo '{"session_id":"s","hook_event_name":"Notification","message":"idle"}' | swarm event notification
exit=0
--- cmd_restore (SessionStart)
$ echo '{"session_id":"s","hook_event_name":"SessionStart","source":"startup"}' | swarm restore
exit=0
```

All four exit 0, silently, no traceback — `is_agent(root, "")` is `bool("") and ...` =
`False`, so `not is_agent` is `True` and they `sys.exit(0)`. Same with a **stale** pane.
The turn is never damaged.

**This attack does not reproduce. The gates are clean on the crash axis.** (They are
still fatal on the *mail-loss* axis — that is A4, a different failure — but the "does it
crash a hook" hypothesis is simply false, and I report that plainly.)

## Summary for structure-red2

**The headline is A2, and it survives the child's §7g correction.** `sender = my_name() or
"operator"` only rescues a root that has never spawned. Every real human spawns, which
names their root, which makes `my_name()` truthy, which makes the fallback dead code —
and from that moment every message the human sends is downgraded from **"the OPERATOR
(the human at the root)"** to **"your parent"** (direct children) or **"another agent"**
(everyone deeper: 5 of 7 in my test tree, and the fraction grows with depth). Pre-patch,
all of them heard "the OPERATOR". `WORLD.md:19` names OPERATOR as one of the four sender
classes; after this fix no ordinary code path can produce it. There is no `--from` and
`operator` is a reserved name, so the human has no way to reclaim their own voice.

**Attack A is real but bounded.** A paneless session claims root-1, root-2, root-3… one
per spawn, forever, and `is_dead`'s new paneless clause makes every phantom render `[live]`
permanently. It hits both versions. But I could not reach `HERDR_ENV=1` + no
`HERDR_PANE_ID` from stock herdr, and I say so rather than dress it up.

**On which code was measured:** the stash contains BOTH bugs the child says it found and
fixed (A3 anonymous sender, A4 mail loss). I verified both against the stash and verified
both are absent from the §7g code. So the artifact on disk is the pre-fix draft, and the
"1 failure" figure cannot have come from it — A4 alone should surface as test errors.
That is red2-mech's call to settle, but my evidence is consistent with the parent's
suspicion.

**Recommended:** the one-line `relation()` change (`sender.startswith("root-")` → OPERATOR)
closes A2 without giving up anything the fix was for. A2 should block the merge.
