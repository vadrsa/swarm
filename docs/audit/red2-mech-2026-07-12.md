## ATTACK B — the `is_dead` hunk. Verdict: **SURVIVES against reality; WOUNDS as a self-inflicted, latent R5 regression.**

```
OLD: def is_dead(a): return live is not None and a.get("pane") not in live
NEW: def is_dead(a): return live is not None and bool(a.get("pane")) and a.get("pane") not in live
```
Consequence: an empty-pane record is now **never dead** → renders live forever, never hidden.

**B.1 — what dead-hiding buys (ruling R5).** `render_ps` docstring / `docs/design/REVIEW.md`:
*"Dead agents render compactly — names only, one shared line (ruling R5): the living tree
stays one screen however long the swarm's history grows."* Measured on a copy of the real
100-record `.swarm`, pure `render_ps`:

```
live= 4 : rendered_lines=  6   dead_hidden= 96
live= 8 : rendered_lines= 10   dead_hidden= 92
live=15 : rendered_lines= 17   dead_hidden= 85
```
R5 collapses 85–96 dead records onto one line. Any record made permanently ineligible for
the `dead:` line defeats it.

**B.2 — against reality: NULL RESULT (SURVIVES).** How many of the real records have an
empty/missing pane?

```
$ for f in .swarm/agents/*.json; do jq -er 'select((.pane//"")=="")|.name' "$f"; done
(zero output — 0 of 100)
```
There is **no existing fleet of paneless records** for the new `is_dead` to resurrect. The
"resurrect the dead into the live tree" arm does **not** reproduce. Stated plainly: this arm
is a null result.

**B.2 — spawn-path zombie: REFUTED (SURVIVES).** Reading `cmd_spawn` end-to-end: the record
is written *after* pane acquisition, and `if not pane: die("herdr tab create failed")` runs
**before** `write_atomic(agent_rec_path(...))`. Simulated with a fake `herdr` whose `tab
create` prints garbage and exits 1: spawn dies, and **no `kid.json` is left on disk** (only
the journal tombstone). A failed tab-create cannot leave a paneless *spawned-agent* record.

**B.2 — where a paneless record DOES come from: `claim_root_name` (WOUNDS, latent).**
`claim_root_name` writes `pane = os.environ.get("HERDR_PANE_ID", "")` with **no non-empty
guard**, while `root_name()` returns `""` immediately when the pane is empty — so it can
never re-find a record it wrote with `pane:""`. With `HERDR_ENV=1` but `HERDR_PANE_ID`
unset, each spawn mints a **new** `root-N` (root-1, root-2, … up to the `die()` at
root-999). Rendered under the new `is_dead`, all of them are live forever:

```
PATCHED ps: 21 lines (10 paneless roots each render live, each nesting a child)
CLEAN   ps: 12 lines (10 roots collapse onto one dead: line, kids flatten under operator)
```
Worse by 9 lines, **+2 lines per additional paneless spawn, unbounded** — the exact
monotonic-growth failure R5 exists to prevent.

**Reachability (honest downgrade).** Real herdr panes *do* carry `HERDR_PANE_ID`
(`w4:p91` — verified in my own env and in §7b). This path needs `HERDR_ENV=1` **and**
`HERDR_PANE_ID` empty, reachable only via `env -u`, an env-scrubbing script/subshell, or an
old pane. I found **no normal in-product path** that produces it. So: **latent /
robustness, not real-today** — but silent (spawn exits 0; phantom roots accrete invisibly)
and it defeats R5. A genuine robustness defect, not a live bug.

**B.3 — the patch's own justification is TRUE, but self-inflicted.** The comment claims that
without the `is_dead` change, a paneless record is "born dead and `eff_parent` reattaches its
live children to `operator` — the flat row, back." The CLEAN column above *is* that flat row.
So the change fixes a real problem — **one the patch itself created** by writing `pane:""`.
It makes self-inflicted paneless records immortal instead of never writing them.

**B.4 — better fix.** Guard `claim_root_name` to refuse when `HERDR_PANE_ID` is empty (die
loud, write nothing — the guard runs before `claim_name`). Then no paneless record can ever
exist, and **the `is_dead` hunk becomes unnecessary and can revert to OLD.** Cost: one small
`die()`; a genuinely broken session (herdr not giving it a pane) fails loudly instead of
silently spawning phantom roots. Exempting only `root-*` in `is_dead` was tested and does
**not** help (doesn't stop the minting, and makes real-pane dead roots immortal too — larger
blast radius). Recommendation: **guard `claim_root_name`, revert `is_dead`.**

## ATTACK C — the hook gates. Verdict: **the O(n) scan and a `load_agents` crash class both WOUND the fix; the spawned-agent hot path SURVIVES.**

The gate moved from `my_name() == "operator"` (pure env read) to a form that, for a **root**
session, calls `my_name()` → `root_name()` → `load_agents()` (a full `os.listdir` +
`json.load` of every record). **Both** the STASH gate (`is_agent`) and the CORRECTED gate
(`my_name() in ("","operator")`) call `load_agents` on the root path — so this attack applies
to both versions.

**C.1 — pricing (BE FAIR).** Timing, mean ms over 40 runs (≈44 ms floor is interpreter
startup):

| case | PATCHED | CLEAN |
|---|--:|--:|
| SPAWNED n=100 | 53.8 | 54.4 |
| SPAWNED n=1000 | 100.2 | 100.8 |
| ROOT n=100 | 60.2 | 43.7 |
| ROOT n=1000 | 201.0 | 43.8 |

ROOT/PATCHED confirmed O(n): 62 ms @100 → 191 ms @1000 → 530 ms @3000.

- **Spawned agent — SURVIVES, genuinely free.** `SWARM_AGENT_ID` is set for every spawned
  agent, so `my_name()`'s `or` short-circuits *before* `root_name()`/`load_agents` — the gate
  adds only one `os.path.exists` stat (STASH) or nothing (CORRECTED). PATCHED ≈ CLEAN at every
  N; the N-scaling that *is* present is the pre-existing delivery scan on **both** variants,
  not the gate. **This is the common, high-frequency case and the attack on it fails. No hit
  manufactured — there isn't one.**
- **Root session — WOUNDS IT.** CLEAN is flat O(1) ~44 ms; PATCHED is O(n) — pure dead
  overhead on the human's own session on every prompt / Stop / SessionStart, on a hook that
  then exits 0 doing nothing. ~150 ms at 1000 agents. (`cmd_deliver` calls `my_name()` twice,
  so a root pays *two* scans per hook.)

**C.2 — can it raise? YES — the non-dict record (headline crash).** `load_agents:150` is
`if r and r.get("name")`. `read_json`'s try/except is *inside* `read_json`; the `.get("name")`
is called in `load_agents`, **outside any try**. A truthy non-dict record (`[1,2,3]`, `"str"`,
`42`) reaches `.get` and raises `AttributeError`. Confirmed directly:

```
$ python3 -c "r=[1,2,3]; r and r.get('name')"
AttributeError: 'list' object has no attribute 'get'
```

Driven through the real hooks as a **true root session** (`env -u SWARM_AGENT_ID`,
`HERDR_PANE_ID=zz:pDEAD`), one `[1,2,3]`-content file in `agents/` (5× each, stable):

```
gate       hook          exit
CLEAN      deliver       0     (clean my_name() is a pure env read; load_agents never runs in the gate)
CLEAN      event stop    0
CLEAN      restore       0
STASH      deliver       1     AttributeError: 'list' object has no attribute 'get'
STASH      event stop    1     AttributeError
STASH      restore       1     AttributeError
CORRECTED  deliver       1     AttributeError
CORRECTED  event stop    1     AttributeError
CORRECTED  restore       1     AttributeError
```

**Caused by the fix (both versions), absent on CLEAN, survives §7g.** It violates the
documented contract (`cmd_deliver`: *"Bulletproof: any error → exit 0, no output, turn
intact"*): the gate's `my_name()` runs *before* the body's `try`. (Rig note: an earlier run
showed a flaky `event stop exit=1` on CLEAN — that was a collision on `HERDR_PANE_ID=w4:p1`,
my *own* live pane, which is in the real live set. Re-run with a dead pane id: CLEAN is
0/5. Reported the corrected, stable result.)

Also confirmed by the Attack-C worker and cross-checked in principle: `agents/` dir mode
`000` → `PermissionError` from `os.listdir` (root, absent on CLEAN); a FIFO record →
`open()` **hangs** forever (root path new, delivery path pre-existing). Survivors (exit 0,
no stray output, both variants): `agents` is a file, broken symlink, symlink loop,
`x.json` is a directory, corrupt/binary/50 MB record, absent/broken `.swarm`, and a
concurrent-writer race (50k renames vs 200 hook runs → 0 nonzero exits; `write_atomic`
tmp+rename is safe).

**The deeper, pre-existing version of the same bug (bigger than the crash).** The same
non-dict record silently **drops a spawned agent's mail** — on CLEAN *and* CORRECTED. The
`cmd_deliver` body (line 718) calls `load_agents` inside its `try`; the poison record raises,
the `try` swallows it, the hook exits 0 with **no injection**, and the message stays queued:

```
spawned agent kid, one queued message, a poison [1,2,3] record elsewhere in agents/:
  CLEAN     poison=no  -> DELIVERED    poison=yes -> LOST
  CORRECTED poison=no  -> DELIVERED    poison=yes -> LOST
```

This is **pre-existing** (not caused by the fix) and worse than the root crash: silent
permanent mail loss, breaking `WORLD.md` *"nothing is ever silently dropped."* The fix
neither causes nor worsens *this* path — it merely adds a *new* root-session surface where
the same `load_agents` fragility surfaces as a loud crash instead of a silent drop.

**One fix closes both.** `if isinstance(r, dict) and r.get("name")` at `load_agents:150`:

```
with the isinstance guard, poison record present:
  spawned mail: DELIVERED (fixed)
  root deliver: exit=0     (fixed, no AttributeError)
  suite:        Ran 80 ... FAILED (failures=1)   <-- still exactly the bug-encoding test
```

One token. Independent of the gate debate. **This is my headline recommendation.**

**Was `is_agent` itself (the STASH gate) a real defect? YES — confirmed independently.**
The gate is `os.path.exists(agents/<name>.json)`. A live agent with a queued message but no
record → the deliver hook exits 0 with empty stdout and the message stays queued:

```
STASH      rec-present -> DELIVERED    rec-absent -> LOST (hook exit 0, empty stdout, never injected)
CORRECTED  rec-present -> DELIVERED    rec-absent -> DELIVERED
```

This reproduces the child `structure-mech`'s own §7d bug report by direct observation: the
`is_agent` gate converts a missing/momentarily-absent record into permanent silent mail
loss. The CORRECTED gate keys on the **name** (`SWARM_AGENT_ID`, always set for spawned
agents) and delivers regardless. **The correction genuinely closes this.** (Note the
`chmod 000` "unreadable record" variant does **not** lose mail on the stash: `os.path.exists`
returns True without reading. Only a truly *absent* record triggers it.)

**C.3 — the `claim_root_name` race.**

- **(i) same pane — WOUNDS IT (TOCTOU, reproduced).** 12 concurrent `swarm spawn` from one
  pane (`HERDR_PANE_ID=SHARED-PANE`), 10 rounds → rounds 2, 5, 7, 8, 9 produced **2–4
  `root-*` records for the one shared pane.** Frozen capture:

  ```
  CAPTURED: 2 root records for pane SHARED-PANE
    root-1 pane='SHARED-PANE' parent=operator
    root-2 pane='SHARED-PANE' parent=operator
  my_name() -> 'root-1'    (sorted-first; root-2 is an orphan phantom root)
  ```
  `claim_name`'s `O_CREAT|O_EXCL` settles the *name* but not the *pane→name* mapping: two
  concurrent spawns both see no root record, both claim different names for the same pane.
  §7c's table claims pane-keying kills the concurrent-root case — but it only reasoned about
  **different** panes. Same pane, concurrent spawn, breaks the "the pane IS identity"
  invariant. Present in both STASH and CORRECTED (`claim_root_name` is identical).
  Reachability: two `swarm spawn` fired from one pane — a script, a `&`-backgrounded pair, a
  fast double-invoke. More reachable than the empty-pane path.

  *(Honesty note: my first same-pane run showed only one root and I nearly called this
  SURVIVES. That run was under-concurrent — a readiness-timeout + a fake launcher that never
  signals serialized the spawns, and a zsh glob-nomatch silently aborted later loop rounds. A
  rebuilt harness with true background overlap reproduces the race. I flag the earlier call as
  a rig artifact and report the corrected result.)*

- **(ii) different panes — SURVIVES.** 20 distinct panes → 20 distinct names, one pane
  each, no collision. `O_CREAT|O_EXCL` guarantees name uniqueness.

## Severity ceiling — stated honestly

The root-session hook **crash** (C.2) and O(n) **cost** (C.1) fire only if a root session's
hook is actually invoked. §7d argues a root session has **no settings file → no hooks
wired**, so in production the gate may never run there. If that holds, the root crash/cost
are largely theoretical *for the hook path* — **but** the underlying `load_agents`
non-dict-unsafety is real on the **spawned-agent mail path** regardless (the pre-existing
silent-drop above), and `load_agents` is now also the fix's `my_name()` hot path. So the
`isinstance` guard is worth shipping on its own merits, independent of whether root hooks
ever fire.

## Recommendations, in priority order

1. **Ship `if isinstance(r, dict) and r.get("name")` at `load_agents:150`.** One token;
   fixes the root-hook crash *and* the pre-existing spawned-mail silent-drop; suite stays at
   the one expected failure. Highest value, lowest risk, orthogonal to every design debate.
2. **Adopt the CORRECTED (§7g) gate over the STASH** — the `is_agent` gate is a real mail-loss
   defect; do not ship the stash. (R0: the stash fails 5 tests; §7g fails 1.)
3. **Guard `claim_root_name` against an empty `HERDR_PANE_ID`** (die loud) — removes the
   paneless-root cause, after which **the `is_dead` hunk can revert to OLD** and R5 is
   preserved without making empty-pane records immortal.
4. **Make the `claim_root_name` pane→name claim atomic** (or serialize per-pane) to close the
   same-pane TOCTOU; and never call `my_name()`/`root_name()` outside a hook's bulletproof
   `try`.
