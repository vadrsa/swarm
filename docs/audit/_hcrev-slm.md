## The load-bearing finding, stated plainly

HARNESS.md sec2.4 asks the right question ("can this model survive this harness at all?") but answers it with a causal claim the evidence doesn't support at the strength claimed. What is MEASURED: two Haiku-pinned agents, in two independent runs, each blocked on a genuine Claude Code tool-permission dialog before reaching any judgment work. What is NOT measured: whether a Sonnet or Opus agent hitting the identical first-touch, no-permissions-block settings file would ALSO block. The source document (`weak-model-delegation-2026-07-13.md`) is honest about this gap — it names the exact falsifier and lists it as the #1 open item — but HARNESS.md's sec2.4/sec4.1/sec7.3/sec8 build a hard spawn-time refusal ("swarm: haiku cannot hold an agent pane: no auto permission mode") on top of a claim its own source document explicitly flags as untested, while HARNESS's own sec8 simultaneously (and correctly) identifies the missing-permissions-block as "a named defect" whose "bill" the Haiku ban represents. Both of these statements cannot be fully true at once: either the wedge is a model property (survivability gate, as sec2.4 claims) or it is an infrastructure defect that would wedge any model on first touch (as sec8 admits) — the doc does not resolve which, and the cheap falsifier that would resolve it was never run.
